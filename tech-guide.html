"프레임워크 없는 단일 HTML 파일" 구조를 쓴 이유 — 이력서/포트폴리오는 "한 번 만들고 끝"인 용도라, 장기 유지보수성보다 휴대성·즉시 실행성이 중요했다. 다른 PC로 옮겨도 npm install 같은 절차 없이 더블클릭으로 열린다. (단, 폴더 참조형이라 capture/ 폴더가 같이 있어야 이미지가 보인다 — 폴더 없이 쓰려면 별도 빌드된 standalone 버전을 쓴다.)

2. HTML 문서 구조 한눈에 보기

<head> <style> ... 전체 CSS (디자인) ... </style> </head> <body> <nav>...</nav> ← 상단 고정 메뉴 (앵커 링크) <section id="hero"> ← 히어로 (핵심 지표 카운트업) <section id="summary"> ← 01 ABOUT (역량 필러) <section id="cases"> ← 02 CASE STUDIES (PAR 구조) <section id="career"> ← 03 CAREER (좌측 레일 타임라인) <section id="ai"> ← 04 AI CAPABILITY (문제/해결) <section id="gallery"> ← 05 GALLERY (썸네일 + 라이트박스) <section id="contact"> ← 06 CONTACT <script> ... 전체 JS (동작) ... </script> </body>

7개 <section>이 위→아래로 쌓인 "단일 페이지 스크롤형(One-page scroller)" 구조다. <nav>의 각 메뉴는 <a href="#career">처럼 섹션 id로 점프하는 앵커 링크다.

3. CSS 문법 설명

3-1. CSS 변수 — :root { --a: ... }

:root{
  --bg:#07101A;       /* 배경 (가장 어두운 톤) */
  --card:#0F2035;     /* 카드 배경 */
  --a:#4A9EBA;        /* 메인 브랜드 컬러 (청록) */
  --a2:#6BBDD6;        /* 강조 텍스트용 밝은 버전 */
  --text:#E8EDF2;     /* 본문 텍스트 */
  --line:#1C3550;     /* 테두리선 */
  --ease:cubic-bezier(0.25,0.46,0.45,0.94); /* 인터랙션 곡선 */
}

--로 시작하는 이름은 CSS 커스텀 프로퍼티(변수)다. var(--a)로 어디서든 불러 쓴다. 색을 한 곳(:root)에서만 바꾸면 전체 페이지 색이 바뀌므로, 디자인 시스템에서 말하는 "디자인 토큰" 패턴이다 (Figma의 컬러 스타일과 같은 개념).

3-2. 반응형 — clamp()

.hero-name{ font-size:clamp(2.6rem, 6vw, 4.8rem); }

clamp(최소, 권장, 최대) — 화면이 좁아지면 6vw(뷰포트 너비의 6%)만큼 줄어들지만 2.6rem보다 작아지지 않고, 커져도 4.8rem을 넘지 않는다. 미디어쿼리 없이도 글자가 화면 크기에 맞춰 자동으로 늘고 줄어든다. 이 문서 전체에서 폰트 크기는 거의 전부 clamp()로 지정되어 있다.

3-3. 카운트업 직전 트랜지션 차단

.num-val[data-count]{ transition:none; }

숫자가 JS로 빠르게 바뀌는 동안 CSS transition이 같이 적용되면 오히려 뚝뚝 끊기듯 보인다. data-count 속성이 붙은 요소는 transition을 꺼서, JS의 requestAnimationFrame 루프가 매끄럽게 그 자리를 대신하게 한다.

3-4. @keyframes — 반복 애니메이션

@keyframes gcGrow{ to{ transform:scaleY(1); } }
@keyframes pfGrow{ to{ width:var(--w); } }

@keyframes는 "정해진 동작을 한 번 또는 반복 재생"할 때 쓴다. 막대 그래프가 0에서 자라나는 효과(gcGrow), 진행률 바가 채워지는 효과(pfGrow) 모두 이 방식이다. --w처럼 CSS 변수를 인라인 스타일로 요소마다 다르게 주면, 같은 @keyframes 하나로 막대마다 다른 최종 길이를 만들 수 있다 (코드 재사용).

4. JavaScript 문법 설명

4-1. IntersectionObserver — "스크롤해서 보일 때" 감지

const revealEls = document.querySelectorAll('.case, .doc, .tl-i, .ai-proj, .promo-item');
revealEls.forEach(el => el.classList.add('reveal'));

const revealObs = new IntersectionObserver(entries => {
  entries.forEach(e => {
    if (e.isIntersecting) { e.target.classList.add('visible'); revealObs.unobserve(e.target); }
  });
}, { threshold: 0.08, rootMargin: '0px 0px -30px 0px' });
revealEls.forEach(el => revealObs.observe(el));

옛날 방식은 scroll 이벤트마다 모든 요소의 위치를 계산해야 해서 느렸다. IntersectionObserver는 브라우저가 "이 요소가 화면에 몇 % 보이는지"를 효율적으로 알려주는 최신 API다. threshold:0.08은 "8% 이상 보이면 알려달라"는 뜻. unobserve로 한 번 보인 요소는 감시를 그만둬서(한 번만 재생), 스크롤할 때마다 다시 계산하지 않아 가볍다.

4-2. requestAnimationFrame — 숫자 카운트업 (easeOutExpo)

function tick(now) {
  const t = Math.min((now - start) / duration, 1);          // 진행률 0~1
  const ease = t === 1 ? 1 : 1 - Math.pow(2, -10 * t);       // easeOutExpo 곡선
  const val = Math.round(ease * target);
  el.textContent = prefix + val + suffix;
  if (t < 1) requestAnimationFrame(tick);
}
requestAnimationFrame(tick);

setInterval보다 requestAnimationFrame이 애니메이션에 더 적합한 이유는, 브라우저의 화면 갱신 주기(보통 초당 60회)에 맞춰 호출되어 더 매끄럽고 배터리 효율적이기 때문이다. 1-Math.pow(2,-10*t)는 "처음엔 폭발적으로 빠르게, 끝엔 서서히 멈추는" easeOutExpo 감속 곡선이다 — 히어로 영역 숫자가 솟구치듯 올라가다 제자리에 착지하는 효과의 핵심. 호버 시 살짝 커지는 scale 효과와 글로우(텍스트 그림자) 감쇠도 같은 진행률(ease) 값을 재사용해서 동시에 계산한다 (값 하나로 여러 효과 동기화).

4-3. 이벤트 위임 — 갤러리 썸네일 클릭

const galleryRoot = document.getElementById('gallery');
galleryRoot.addEventListener('click', function (e) {
  const thumb = e.target.closest('.promo-thumb');
  if (!thumb) return;
  const galId = thumb.dataset.galId;
  const idx = parseInt(thumb.dataset.imgIdx, 10);
  openGal(galId, idx);
});

썸네일 하나하나에 클릭 이벤트를 따로 붙이지 않고, 갤러리 전체를 감싸는 부모(#gallery) 하나에만 리스너를 건다. 클릭이 일어나면 e.target.closest('.promo-thumb')로 "클릭된 지점에서 가장 가까운 썸네일 요소"를 찾아 올라간다 — 이를 이벤트 위임(event delegation)이라 부른다. 썸네일이 나중에 추가되거나 삭제돼도 리스너를 다시 걸 필요가 없다는 장점이 있다. 어떤 이미지를 열지는 HTML의 data-gal-id/data-img-idx 커스텀 속성에 저장해두고 JS가 그 값만 읽어서 판단한다.

4-4. 라이트박스 데이터 구조

const galleries = {
  g1: [
    {src:'./capture/g/gal-01.jpg', cap:'캡션 1/4'},
    {src:'./capture/g/gal-02.jpg', cap:'캡션 2/4'}
  ],
  g2: [ ... ]
};
let curGal = [], curIdx = 0;

갤러리 ID(g1, g2...)를 key로, 그 안의 이미지 배열을 value로 갖는 단일 객체에 모든 갤러리 데이터를 모아둔다. 썸네일을 클릭하면 해당 갤러리 배열을 curGal에 담고, 화살표 키(←/→)나 버튼을 누르면 curIdx만 ±1 해서 같은 함수(showLb)로 다음/이전 이미지를 그린다 — "현재 보고 있는 위치"를 변수 2개로만 관리하는 단순한 상태 관리 패턴이다.

5. 폴더 구조 (개발용 원본 기준)

portfolio/ ├─ index.html ← 실제 포트폴리오 (폴더 참조형, 개발·수정용) ├─ portfolio_standalone.html ← 이미지 Base64 내장 단일 파일(이력서 업로드용) ├─ capture/ ← 작업물 원본 캡처 (한글/공백 파일명 보존) └─ capture/g/ ← ASCII 안전 이름(gal-01~43)으로 복사한 참조용 사본

수정은 항상 index.html(폴더 참조형)에서 먼저 하고, 끝나면 standalone 버전을 재생성하는 순서를 지킨다 — 역순으로 작업하면 다음 수정 때 변경 내용이 사라진다.
